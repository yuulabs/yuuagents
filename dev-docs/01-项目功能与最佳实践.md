# yuuagents：项目功能与最佳实践

## 功能概览

- Linux service 形态的 daemon：常驻进程管理多个任务，提供 HTTP API（Unix Domain Socket）。
- CLI 一等公民：所有核心能力都通过 `yagents` 命令完成；CLI 是 daemon 的 thin client。
- Agent 运行时：一个有状态对话实体，执行 “LLM → 工具 → LLM” step loop。
- 工具体系：基于 `yuutools` 的 `ToolManager` 与 `depends()` 注入；内置工具覆盖 bash / file / web。
- Docker 隔离：所有 bash/file 工具在容器中执行；宿主机根目录只读挂载到 `/mnt/host`。
- Skills（Agent Skills 标准）：daemon 启动/扫描时发现 `SKILL.md`，把元信息注入 system prompt。
- 成本与用量统计：通过 `yuutrace` 记录 usage/cost，并回写到 agent 状态。
- 配置：YAML 配置（`~/.yagents/config.yaml`），支持运行时热重载（不重启 daemon）。

## 命令速览

| 场景 | 命令 | 说明 |
|---|---|---|
| 安装依赖 | `uv sync` | 初始化/同步虚拟环境依赖 |
| 一次性安装/初始化 | `uv run yagents setup` | 写入配置、创建目录、准备 Docker、注册 systemd user service |

| 列出任务 | `uv run yagents list` | 列出 daemon 当前记录的任务摘要 |

| 运行测试 | `uv run pytest` | 运行测试 |

## 详细用法与注意事项

### 基础原则（建议作为默认工作流）

- 先用 `yagents setup` 生成/安装 `~/.yagents/config.yaml`，再启动 daemon；不要手工从零写配置。
- 把 API Key 放到环境变量（由 `providers.<name>.api_key_env` 指定），不要写进配置文件。
- 容器内的可写工作区是 `/home/yuu`（默认）；宿主机只读挂载是 `/mnt/host`。
- 如果任务需要读宿主机代码/文件，按路径读取 `/mnt/host/...`；需要写结果请写到 `/home/yuu/...`。

### `yagents setup`

- 参数
  - `--config PATH`：完全替换默认配置（需要确认）。
  - `--overrides PATH`：以 deep-merge 的方式叠加覆盖项。
  - `--project-dir PATH`：显式指定包含 `config.example.yaml` 的项目根目录（默认自动向上查找）。
- 行为要点
  - 默认会从项目根的 `config.example.yaml` 开始，并自动叠加 `config.overrides.yaml`（如果存在）。
  - 会创建 `~/.yagents/skills`、`~/.yagents/dockers` 以及 `skills.paths` 指向的目录。
  - 会尝试 pull 配置中的 Docker image；如果 Docker 不可用，会给出提示但不强行失败。

### `yagents start`

- 参数
  - `--config PATH`：从指定路径加载配置，否则使用 `~/.yagents/config.yaml`。
- 运行模式
  - 前台启动（适合配合 systemd 管理）。
  - 监听 Unix Domain Socket（默认 `~/.yagents/yagents.sock`）。

### `yagents run`

- 参数
  - `--agent NAME`：选择 `config.yaml` 里 `agents.<NAME>`（默认 `main`）。
  - `--task TEXT`：必填，任务描述。
  - `--persona TEXT`：覆盖 agent 配置里的 persona/system prompt。
  - `--tools a,b,c`：逗号分隔的工具名列表（覆盖默认 tools）。
  - `--skills a,b,c`：逗号分隔的 skill 名列表（覆盖默认 skills）。
  - `--model NAME`：覆盖模型。
  - `--container ID`：复用已有容器（优先使用该容器）。
  - `--image NAME`：为该任务创建新容器（与 `--container` 互斥）。
- quirk
  - 未知的工具名会被静默忽略（只会注册已知内置工具）。如果需要强约束，应在上层做校验。

### `yagents list / status / logs`

- `list`：输出更适合人读的摘要行（状态、task_id、agent_id、steps、cost、task 前缀）。
- `status <task_id>`：输出 JSON，适合脚本消费（包括 error/cost/steps/last_assistant_message）。
- `logs <task_id>`：把 history 按 role 输出；tool_call 结果以 JSON 形式打印。

### `yagents input`

- 适用场景：工具链里存在等待用户输入的环节（例如某些交互式流程）。
- 行为：向该任务的 `input_queue` 写入一条字符串消息，供工具读取。

### `yagents stop-agent`

- 行为：取消运行中的 asyncio task，并将 agent 状态置为 `cancelled`。
- 建议：如果任务使用了 `--image` 创建临时容器，取消后应确认容器是否清理（当前清理由 DockerManager 统一 stop 时做一次性清理）。

### `yagents skills list / scan`

- `list`：返回 daemon 当前缓存的 skills 元信息。
- `scan`：重新扫描 `skills.paths` 目录；只读取每个 skill 目录下的 `SKILL.md`。
- quirk
  - frontmatter 解析是最小实现：仅支持 `name:` 与 `description:` 的简单行格式；复杂 YAML 可能无法识别。
  - frontmatter 必须出现在文件开头（严格匹配 `--- ... ---`）。

### `yagents config`

- 无参数：打印当前磁盘配置（`~/.yagents/config.yaml` 解析后的摘要），不是从 daemon 内存直接读取。
- `--overrides FILE` / `--config FILE`：更新磁盘配置，并尝试调用 daemon 的 `/api/config/reload` 做热重载。

### `yagents stop / down`

- `stop`：当前实现不向 daemon 发送 shutdown，仅做 health 检查并提示如何用 systemd 停止。
- `down`：卸载 systemd user service（停止并注销）。

## 内置工具速览（给 agent 的 “能力”）

| 工具名 | 参数 | 主要用途 |
|---|---|---|
| `execute_bash` | `command`, `timeout` | 在容器内执行命令（默认 timeout=120s，最大 600s） |
| `read_file` | `path` | 读取容器内文件内容 |
| `write_file` | `path`, `content` | 写入容器内文件（自动创建父目录） |
| `delete_file` | `path` | 删除容器内文件 |
| `web_search` | `query`, `max_results` | Tavily 搜索并返回 LLM 友好文本 |

## 需要评估的点（标记，不改动）

- daemon 生命周期可能被重复管理：`server.serve()` 里显式 `manager.start()`，而 Starlette lifespan 里也会 `manager.start()`。
- `yagents stop` 未实现真正的 shutdown（仅提示）。
- 文档漂移风险：`design/design.md` 描述的命令与配置格式（TOML、dashboard 等）与当前实现（YAML、无 dashboard 命令）不一致。
- `write_file` 使用 heredoc 分隔符 `YUUEOF`：如果内容中出现同名整行标记，可能导致截断。
