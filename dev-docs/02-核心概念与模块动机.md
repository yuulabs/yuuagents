# 核心概念与模块动机

## 1. 运行模型

`yuuagents` 当前不是“单个 Agent 类”包，而是一个围绕任务执行、工具注入、持久化和 daemon 调度拆开的运行时。

主线可以概括成：

1. `AgentConfig` 描述一次 agent 运行需要什么
2. `AgentContext` 描述这次运行处在什么上下文里
3. `Session` 持有活跃运行的状态
4. `core.flow.Agent` 执行真正的 step loop
5. `AgentManager` 在 daemon 侧负责调度、能力注入和持久化

SDK 路径和 daemon 路径共享同一套会话模型，只是前者由 `LocalAgent` 包一层，后者由 HTTP daemon 包一层。

## 2. 模块边界

### `src/yuuagents/config.py`

这里定义了结构化配置和默认值：

- `DaemonConfig`
- `DbConfig`
- `YuuTraceConfig`
- `SnapshotConfig`
- `DockerConfig`
- `TavilyConfig`
- `ProviderConfig`
- `AgentEntry`
- `Config`

为什么用 `msgspec.Struct`：

- 配置读取和序列化都很直接
- 默认值清晰
- 验证和转换成本低

这里还提供了：

- `load_yaml(...)`
- `load_packaged_default_yaml()`
- `load(...)`
- `load_packaged_default()`
- `_deep_merge(...)`
- `find_project_root(...)`

实际安装默认值来自 `src/yuuagents/config.default.yaml`，不是源码里 `Config()` 的裸默认值。

### `src/yuuagents/agent.py`

`AgentConfig` 只描述一轮运行的不可变配置：

- `agent_id`
- `tools`
- `llm`
- `system`
- `tool_batch_timeout`

这个对象是 runtime config，不是全局项目配置。

### `src/yuuagents/context.py`

`AgentContext` 描述运行上下文，包括：

- `task_id`
- `agent_id`
- `workdir`
- `capabilities`
- `session`
- `delegate_depth`

它是 capability 注入和 session 关联的落点。`delegate_depth` 用来限制递归委派深度。

### `src/yuuagents/runtime_session.py`

`Session` 是 host-facing 的活跃运行包装器，负责：

- 持有 `AgentConfig` 和 `AgentContext`
- 暴露 `status`、`steps`、`total_cost_usd` 等读接口
- 维护 `history`
- 包装 `snapshot()`、`kill()`、`step_iter()`

这个层的作用是把底层 flow 封成宿主可操作对象，而不是让外部代码直接碰内部实现细节。

### `src/yuuagents/local.py`

`LocalAgent` 是 SDK 路径的薄封装。

它做的事情很少：

- 把传入的工具列表转换成 `yt.ToolManager`
- 组装 `AgentContext`
- 创建 `Session`
- 提供 `start()` / `run()` / `run_once()`

`run_once(...)` 只是 `LocalAgent.create(...).run(...)` 的便利函数。

### `src/yuuagents/core/flow.py`

这里是实际的 agent 执行循环、flow tree、snapshot 逻辑和工具调用编排的核心实现。

对外层来说，最重要的是：

- `snapshot()` 返回可恢复的 `AgentState`
- `kill()` 会停止当前 flow
- `messages` 和 `steps` 是 flow 的外显状态

### `src/yuuagents/daemon/manager.py`

`AgentManager` 是 daemon 侧的总控对象，负责：

- 维护 task registry
- 创建 root session 和 delegated child session
- 从配置和环境变量里注入 capability
- 构造系统提示词
- 持久化任务元数据和 snapshot
- 恢复未完成任务

这里有一个关键设计点：daemon 并不是“额外一层壳”，而是负责把 capability 和任务状态补齐的运行主机。

### `src/yuuagents/daemon/server.py`

这里把 daemon 变成一个 UDS HTTP server，并且顺带拉起 yuutrace server。

当前启动顺序是：

1. 确保 yuutrace server 可用
2. 初始化 tracing endpoint
3. 初始化 `AgentManager`
4. 绑定 Unix socket，启动 Starlette/Uvicorn

### `src/yuuagents/daemon/api.py`

这里定义 HTTP 路由：

- `GET /health`
- `POST /api/agents`
- `GET /api/agents`
- `GET /api/agents/{task_id}`
- `GET /api/agents/{task_id}/history`
- `POST /api/agents/{task_id}/input`
- `DELETE /api/agents/{task_id}`
- `GET /api/config`
- `POST /api/config/reload`

### `src/yuuagents/persistence.py`

这里实现 SQLite 持久化、snapshot checkpoint 和恢复读取。

### `src/yuuagents/tools/*`

这里是 built-in tools 的实现层，按 capability 分成：

- 永久在线工具
- delegate/后台控制工具
- Docker 工具
- web 工具

## 3. 为什么这样拆

### 配置和运行时分离

配置层只负责“描述系统应该长什么样”，运行时对象负责“现在正在发生什么”。

这样做的结果是：

- `Config` 可以被序列化、写盘、热重载
- `Session` 和 `AgentManager` 可以承载活跃状态
- SDK 路径和 daemon 路径共享配置格式

### SDK 和 daemon 共享同一套 flow

SDK 路径不应该再实现一套执行引擎，所以 `LocalAgent` 只负责把最小上下文搭好，然后交给 `Session` 和 `core.flow.Agent`。

daemon 路径也不应该自己重写一套 agent loop，所以 `AgentManager` 只负责会话工厂、capability 注入、持久化和恢复。

### capability 是显式的

工具是否能执行，不靠名字猜，而靠 `AgentCapabilities` 注入。

这让下面这些事情都变得可控：

- Docker 相关工具不会在没有 Docker 时假装可用
- `delegate` 不会在没有 host delegate manager 时误工作
- `web_search` 不会在没有 Tavily key 时误调用

### snapshot 和 tracing 分开

snapshot 负责恢复，tracing 负责可观测性。

两者是不同的数据平面，不应该混成一个概念。

## 4. 当前实现中的几个约束

- `AgentManager` 只在 daemon 侧存在
- `LocalAgent` 不会自动给你 Docker 或 delegate capability
- `install` 写入的是包内默认模板叠加后的结果，不是任意源码文件
- `up` 会拒绝切换到新的数据库路径，以免把旧数据库留在磁盘上
- `Config.validate()` 目前是辅助检查，不代表所有错误都会在命令入口处被硬阻断

## 5. 模块映射

如果要快速定位一个问题，优先看这些文件：

- 配置和默认值：`src/yuuagents/config.py`
- SDK 路径：`src/yuuagents/local.py`、`src/yuuagents/runtime_session.py`
- 运行主循环：`src/yuuagents/core/flow.py`
- daemon 路径：`src/yuuagents/daemon/server.py`、`src/yuuagents/daemon/api.py`、`src/yuuagents/daemon/manager.py`
- snapshot 与任务表：`src/yuuagents/persistence.py`
- 工具实现：`src/yuuagents/tools/`
